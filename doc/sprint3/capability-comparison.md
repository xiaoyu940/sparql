# RS Ontop Core vs 开源 Ontop 能力对比

> 更新时间: 2026-03-29  
> 对比版本: RS Ontop Core Sprint3 vs Ontop (官方开源版本)

---

## 1. 核心架构对比

| 能力项 | RS Ontop Core (当前) | 开源 Ontop | 差距分析 |
|--------|---------------------|-----------|---------|
| **开发语言** | Rust + pgrx | Java | 语言差异，Rust 有内存安全优势 |
| **部署形式** | PostgreSQL 扩展 | 独立服务 + 可选嵌入 | RS Ontop 与 PG 深度集成 |
| **查询引擎** | 自定义 IR + SQL 生成 | Apache Calcite 优化器 | Ontop 优化器更成熟 |
| **SPARQL 支持** | 1.1 基础 (SELECT, FILTER, OPTIONAL, UNION) | 1.1 完整 + 扩展 | RS Ontop 缺少部分高级特性 |
| **R2RML 支持** | 部分实现 (映射定义) | 完整 R2RML + OWL | RS Ontop 映射功能较简单 |

## 8. RS Ontop Core 优势

1. **PostgreSQL 原生集成**: 作为 PG 扩展，无需额外部署
2. **内存安全**: Rust 语言保证，避免内存泄漏和崩溃
3. **性能**: 零拷贝设计，Rust 编译优化
4. **部署简单**: 单二进制文件，无 JVM 依赖
5. **流式处理**: 支持大结果集的流式返回

## 10. 建议发展路线

### Sprint 4 (短期)
- 完善 SPARQL 函数支持
- 实现子查询
- 优化器补强 (投影归一化、列剪枝)

### Sprint 5 (中期)
- 属性路径支持
- 查询缓存
- 性能基准测试与调优

### Sprint 6+ (长期)
- 可选的 OWL 推理支持
- 联邦查询能力
- 其他数据库方言适配

---

**结论**: RS Ontop Core 在核心 SPARQL 查询和 PostgreSQL 集成方面已达到可用状态，但在高级 SPARQL 特性、查询优化和推理能力方面与开源 Ontop 仍有差距。建议优先完善子查询和优化器，再逐步添加高级特性。
