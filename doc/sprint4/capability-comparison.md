# RS Ontop Core vs 开源 Ontop 能力对比

> 更新时间: 2026-03-29  
> 对比版本: RS Ontop Core Sprint4 vs Ontop (官方开源版本)  
> 基于 Sprint3 完成后实际代码状态

---

## 1. 核心架构对比

| 能力项 | RS Ontop Core (当前) | 开源 Ontop | 差距分析 |
|--------|---------------------|-----------|---------|
| **开发语言** | Rust + pgrx | Java | 语言差异，Rust 有内存安全优势 |
| **部署形式** | PostgreSQL 扩展 | 独立服务 + 可选嵌入 | RS Ontop 与 PG 深度集成 |
| **查询引擎** | 自定义 IR + SQL 生成 | Apache Calcite 优化器 | Ontop 优化器更成熟 |
| **SPARQL 支持** | 1.1 扩展 (聚合、子查询、BIND) | 1.1 完整 + 扩展 | RS Ontop 接近完整支持 |
| **R2RML 支持** | 部分实现 (映射定义) | 完整 R2RML + OWL | RS Ontop 映射功能较简单 |

## 8. 综合评估

### 8.1 与开源 Ontop 对比

| 维度 | Sprint3 前 | Sprint4 当前 | 目标 |
|------|-----------|-------------|------|
| **SPARQL 完整度** | 50% | 75% | 90% |
| **查询优化** | 60% | 80% | 90% |
| **推理能力** | 10% | 10% | 50% |
| **数据源支持** | 20% | 20% | 40% |
| **生产就绪度** | 40% | 65% | 85% |

### 8.2 核心优势

1. **PostgreSQL 原生集成** - 作为扩展部署，无需额外服务
2. **内存安全** - Rust 语言保证
3. **流式查询** - 支持大结果集
4. **聚合能力** - 已对齐主流需求

### 8.3 核心差距

1. **推理能力** - 无 OWL/RDFS 支持
2. **多数据源** - 仅 PostgreSQL
3. **联邦查询** - 无 SERVICE 支持
4. **属性路径** - 复杂导航模式不支持

---

## 9. 结论

**Sprint4 状态**: 基础 SPARQL 查询能力（聚合、子查询、BIND、VALUES）已完成，与开源 Ontop 的**基础查询能力差距缩小至 25%**。

**建议 Sprint4 重点**:
1. 完善 ORDER BY/OFFSET 的 SQL 生成（低工作量，高价值）
2. 添加查询缓存层（性能优化）
3. 扩展 SPARQL 函数库（兼容性提升）
